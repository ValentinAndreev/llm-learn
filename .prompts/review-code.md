# Прайминг: Review / Verify

## Контекст для агента перед ревью кода

Прочитай:
1. Spec + plan активной фичи (в `memory_bank/features/<id>_<name>/`)
2. `memory_bank/engineering/conventions.md` — coding style, testing rules

Сохрани результат ревью в `memory_bank/features/<id>_<name>/reviews/impl.md`.
Если это повторное ревью после исправлений, перезапиши тот же файл; историю сохранит git.

---

## Fail-Fast Preconditions

- Для `review: <id>` обязателен feature package с существующими `spec.md` и `plan.md`.
- Если одного из файлов нет, остановись и верни blocker-сообщение.
- Пример:
  `BLOCKER: missing memory_bank/features/<id>_<name>/plan.md. Cannot run review: <id>. Next step: create or restore the Plan first.`
- Не подменяй отсутствующие ограничения анализом одного только кода или диффа.


## Быстрый чеклист перед Done

```bash
bundle exec rspec                    # все тесты зелёные
bundle exec rubocop                  # lint
bundle exec brakeman --no-pager      # security (Ruby)
bin/bundler-audit                    # уязвимости в gems
bin/importmap audit                  # уязвимости в JS deps
bundle exec steep check              # типизация
```

Фича Done только если всё выше зелёное локально.

## Code Review (semi-formal)

Ты — строгий code reviewer. Нужен semi-formal reasoning по изменению.
Не давай общих фраз. Каждое утверждение подкрепляй ссылкой на конкретный код.

Контекст:
- Цель изменения: <вставить>
- Ограничения: <вставить>
- Дифф/файлы: <вставить>

Выведи ответ строго в формате:

**1) Предпосылки**
- Явные допущения, без которых выводы недействительны.

**2) Инварианты и контракты**
- Какие свойства системы должны сохраняться после изменения.

**3) Трассировка путей выполнения**
- Ключевые happy-path и error-path.
- Где поведение изменилось, где не изменилось.

**4) Риски и регрессии**
- Список рисков (severity: high/medium/low).
- Для каждого: почему риск реален и где в коде это видно.

**5) Вердикт по эквивалентности поведения**
- Эквивалентно / Неэквивалентно / Недостаточно данных.
- Если неэквивалентно: минимальный контрпример.

**6) Что проверить тестами**
- Топ-5 проверок, которые закроют неопределённость.

**7) Confidence**
- Оценка 0..1 и коротко: что мешает дать более высокий confidence.

Правила: не придумывай факты которых нет в диффе/коде; если данных не хватает — помечай "Недостаточно данных".
