# Сохранение диалогов

**Brief:** `memory_bank/features/001_save_dialogs/brief.md`

## Цель

Сохранять диалоги с LLM в БД, чтобы пользователь мог вернуться к любому из них и продолжить разговор.

## Scope

- Входит (намеренный вертикальный срез через все слои; намеренно затрагивает 6 компонентов вместо рекомендуемых ≤3 — разбивка нецелесообразна из-за тесной связанности; итого 6: `Dialog`, `Message`, `ChatChannel`, `ChatController`, `DialogsController`, Stimulus-контроллер):
  - Модели `Dialog` и `Message` с миграциями
  - Автосоздание нового диалога при первом сообщении в сессии
  - Сохранение каждого сообщения (роль + контент) в БД — сообщение пользователя до отправки в LLM, ответ ассистента после получения
  - Передача истории сообщений диалога в LLM как контекст при продолжении
  - Отображение истории в UI при открытии диалога
  - Dropdown со списком диалогов (выбор, переименование, удаление)
  - Переименование диалога через inline-редактирование названия
  - Каскадное удаление сообщений при удалении диалога
- НЕ входит:
  - Аутентификация и разграничение диалогов по пользователям
  - Поиск и фильтрация по диалогам
  - Пагинация списка диалогов
  - Стриминг ответа LLM по частям

## Требования

1. При отправке первого сообщения в сессии автоматически создаётся новый `Dialog`. Название диалога формируется из первых 50 символов первого сообщения пользователя.
2. Каждое сообщение пользователя и ответ LLM сохраняются как `Message` с полями `role` (`user` / `assistant`) и `content`.
3. При открытии существующего диалога все его сообщения загружаются в UI в хронологическом порядке.
4. При открытии существующего диалога его история передаётся в LLM как контекст — новые сообщения продолжают разговор.
5. Dropdown отображает все диалоги из БД, отсортированные по дате последнего сообщения (новые сверху).
6. Пользователь может переименовать диалог: название редактируется inline (двойной клик или кнопка рядом с названием), сохраняется по потере фокуса или нажатию Enter, отменяется по Escape. Пустое название не сохраняется.
7. Удаление диалога через dropdown удаляет его вместе со всеми сообщениями. После удаления текущего открытого диалога интерфейс сбрасывается: чат-лента очищается, заголовок диалога исчезает, поле ввода разблокируется, `currentDialogId` сбрасывается в `null`.
8. Если диалогов нет, dropdown показывает состояние «Нет сохранённых диалогов».
9. При открытии существующего диалога (`open_dialog`) input отправки сообщения блокируется немедленно и разблокируется только после получения `type: "history"` от сервера.
10. Ошибка сохранения в БД не должна ломать UX — в `ChatChannel#receive` исключение `ActiveRecord::RecordInvalid` / `ActiveRecord::StatementInvalid` перехватывается в `rescue`, в канал бродкастится `{ type: "error", message: "Не удалось сохранить сообщение" }`, Stimulus отображает его как flash.
11. При ошибке загрузки истории (`open_dialog` завершился неудачей) пользователь видит flash-уведомление об ошибке, UI остаётся в текущем состоянии, input разблокируется.
12. При ошибке удаления диалога (сетевая или серверная) пользователь видит flash-уведомление об ошибке, диалог остаётся в dropdown.
13. При ошибке переименования диалога (сетевая, серверная или ошибка валидации) пользователь видит flash-уведомление об ошибке, название возвращается к прежнему значению.

## Инварианты

- `Message` всегда принадлежит `Dialog` (`dialog_id NOT NULL`)
- `Dialog#title` — не может быть пустой строкой или `nil`
- `Message#role` — строго `"user"` или `"assistant"`, других значений быть не может
- Удаление `Dialog` всегда каскадно удаляет все его `Message`
- `Dialog` и первое `Message` создаются атомарно в одной транзакции — либо оба сохранены, либо ни одного
- Существующие тесты `ChatChannel` проходят без изменений

## Acceptance Criteria

- [ ] Новый диалог создаётся в БД при отправке первого сообщения
- [ ] Название диалога = первые 50 символов первого сообщения пользователя
- [ ] Каждое сообщение пользователя сохраняется в БД с `role: "user"` до отправки в LLM
- [ ] Каждый ответ LLM сохраняется в БД с `role: "assistant"` после получения
- [ ] Открытый диалог отображает все сообщения в UI в порядке `created_at ASC`
- [ ] LLM-клиенту передаётся массив `messages` диалога в порядке `created_at ASC` перед новым сообщением пользователя
- [ ] Dropdown содержит все диалоги, отсортированные по дате последнего сообщения (новые сверху)
- [ ] При отсутствии диалогов dropdown показывает «Нет сохранённых диалогов»
- [ ] Двойной клик на названии диалога переводит его в режим редактирования
- [ ] Название сохраняется в БД по Enter или потере фокуса
- [ ] Escape отменяет редактирование, возвращая прежнее название
- [ ] Пустое название не сохраняется — поле возвращается к текущему значению
- [ ] Удаление диалога удаляет запись и все связанные `Message` из БД
- [ ] После удаления текущего диалога: чат-лента очищается, заголовок исчезает, input разблокирован, `currentDialogId` = `null`
- [ ] Пока история диалога загружается, input отправки сообщения заблокирован (disabled)
- [ ] При ошибке сохранения в БД пользователь видит flash-уведомление об ошибке
- [ ] При ошибке загрузки истории пользователь видит flash-уведомление об ошибке, UI остаётся в текущем состоянии, input разблокируется
- [ ] При ошибке удаления диалога пользователь видит flash-уведомление об ошибке, диалог остаётся в dropdown
- [ ] При ответе сервера 422 на PATCH /dialogs/:id название возвращается к прежнему значению и показывается flash-уведомление об ошибке
- [ ] При ошибке переименования пользователь видит flash-уведомление об ошибке, название возвращается к прежнему значению
- [ ] При сбое `GET /dialogs` после получения `dialog_created` пользователь видит flash-уведомление об ошибке, dropdown не обновляется
- [ ] Все существующие тесты проходят
- [ ] Инварианты не нарушены

## Ограничения

- Архитектура: стандартный Rails MVC, без service objects
- Взаимодействие с LLM остаётся в `ChatChannel`, логика работы с БД — там же или в моделях
- Не добавлять новые гемы
- Не трогать существующие миграции
- Типизация: добавить RBS-сигнатуры для новых моделей и методов
- Скоуп: все 6 компонентов (`Dialog`, `Message`, `ChatChannel`, `ChatController`, `DialogsController`, Stimulus-контроллер) — единый атомарный вертикальный срез; дробить на подфичи нельзя
