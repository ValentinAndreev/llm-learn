# Поля workflow для диалога обучения

**Brief:** `memory_bank/features/003_add_dialog_learning_workflow_fields/brief.md`

## Цель

Сделать так, чтобы диалог хранил своё состояние как процесс работы над одной темой и не терял его между сессиями.

## Scope

- Входит (3 компонента: миграция/schema, `Dialog`, совместимость текущего chat flow):
  - Новые поля в `dialogs`
  - Валидации и правила значений в модели `Dialog`
  - Совместимость существующих сценариев создания, открытия, переименования и удаления диалога
- НЕ входит:
  - История версий prompt
  - Хранение brief
  - Новый UI для редактирования этих полей

## Требования

1. В таблицу `dialogs` должны быть добавлены поля: `goal:text`, `workflow_state:string`, `topic_slug:string`, `current_conspect_prompt:text`.
2. Поле `workflow_state` должно быть `NOT NULL` и иметь значение по умолчанию `collecting_info`.
3. Допустимые значения `workflow_state`: `collecting_info`, `intake_complete`, `brief_ready`, `prompt_ready`, `conspect_generated`, `conspect_reviewed`, `conspect_needs_revision`.
4. Для существующих записей `Dialog` значение `workflow_state` должно быть заполнено как `collecting_info`; остальные новые поля должны остаться `NULL`.
5. Значение `goal` может быть `NULL`, но если оно задано, то не может сохраняться как пустая строка после `strip`.
6. Значение `topic_slug` может быть `NULL`, но если оно задано, то должно соответствовать шаблону `\A[a-z0-9]+(?:-[a-z0-9]+)*\z`.
7. Поле `current_conspect_prompt` может быть `NULL` до шага сборки итогового prompt и может быть перезаписано последующими фичами.
8. Существующие сценарии чата не должны требовать заполнения новых полей: отправка сообщения, загрузка истории, переименование и удаление диалога должны продолжать работать при `NULL` в новых колонках.
9. При попытке сохранить недопустимый `workflow_state` или недопустимый `topic_slug` запись `Dialog` должна быть невалидной.

## Инварианты

- У каждого `Dialog` всегда есть допустимое значение `workflow_state`
- `Dialog#title` и существующая логика сообщений не меняются
- Таблица `messages` и её контракт не меняются
- `topic_slug` не может быть заполнен невалидным slug

## Acceptance Criteria

- [ ] В `dialogs` добавлены поля `goal`, `workflow_state`, `topic_slug`, `current_conspect_prompt`
- [ ] У `workflow_state` есть значение по умолчанию `collecting_info`
- [ ] Существующие диалоги получают `workflow_state = collecting_info`
- [ ] Модель `Dialog` не даёт сохранить недопустимый `workflow_state`
- [ ] Модель `Dialog` не даёт сохранить невалидный `topic_slug`
- [ ] Пустое значение `goal` не сохраняется как строка из пробелов
- [ ] Существующие сценарии `ChatChannel` работают при `NULL` в новых полях
- [ ] Переименование и удаление диалога продолжают работать без изменений
- [ ] Все существующие тесты проходят
- [ ] Инварианты не нарушены

## Ограничения

- Использовать стандартные Rails migration/model validations
- Не добавлять новые endpoints
- Не менять контракт `Message`
- Если меняется публичный API модели `Dialog`, обновить RBS
