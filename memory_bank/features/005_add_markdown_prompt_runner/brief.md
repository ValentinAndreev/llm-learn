## Цель

Сейчас каждый новый LLM-сценарий требует дублирования кода вызова модели в `ChatChannel` или соседнем коде. Нужен единый механизм запуска templates, чтобы добавление нового сценария не разрастало channel-логику и не требовало копирования вызовов.

## Для кого

Для разработчика проекта, чтобы дальнейшие сценарии intake, build_prompt и self_review не были размазаны по `ChatChannel` строками и условными ветками.

## Доменный контекст

Система будет иметь несколько LLM-сценариев вокруг одного диалога: проверка полноты, сборка prompt, генерация конспекта, self-review.

## Технический контекст

Сейчас вызов LLM идёт напрямую через `RubyLLM.chat` в `ChatChannel`. Для следующих фич нужен переиспользуемый runner поверх markdown templates.

## Требования

- Механизм должен запускать LLM-сценарий по markdown template из согласованного каталога.
- Механизм должен принимать входные данные для подстановки в template.
- Вызов модели должен идти через текущую интеграцию с RubyLLM.
- Ответ LLM должен быть доступен вызывающему коду для дальнейшей обработки; внутреннее представление определяется реализацией.
- При отсутствии template или ошибке вызова LLM механизм должен возвращать явную ошибку вызывающему коду.

## Что НЕ в скоупе

- Асинхронные jobs
- Usage/cost logging
- Multi-provider routing
