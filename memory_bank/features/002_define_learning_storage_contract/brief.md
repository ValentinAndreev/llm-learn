## Цель

Сейчас правила хранения prompt templates и результатов по темам нигде не зафиксированы, поэтому каждая следующая фича рискует сохранять файлы по-своему. Нужен единый файловый контракт, по которому можно реализовывать и проверять хранение без дополнительных уточнений.

## Для кого

Для разработчика проекта, который будет реализовывать file-first хранение учебных артефактов и промптов.

## Доменный контекст

Диалоги и сообщения уже сохраняются в БД и остаются сессионным слоем. Учебные артефакты по теме должны храниться на диске в markdown, а не в отдельных CRUD-сущностях БД.

## Технический контекст

Текущий проект — Rails + ActionCable + Stimulus. В репозитории уже есть процесс через `memory_bank/features/<id>/{brief,spec,plan}.md`, но для runtime-данных тем и prompt templates контракт хранения пока не определён. Рабочая гипотеза на текущем этапе — использовать отдельное файловое пространство для prompt templates и результатов по темам.

## Требования

- Должен быть задокументирован единый файловый контракт, которому можно следовать без дополнительных уточнений.
- Должно быть явно описано, какие данные остаются в БД, а какие хранятся в файловой структуре.
- Должен быть зафиксирован момент, когда у конкретной темы появляется файловый каталог.
- Должен быть перечислен обязательный набор артефактов первой успешной генерации темы.
- Должны быть задокументированы правила именования и идентификации темы.
- Должен быть зафиксирован способ хранения метаданных артефактов: либо в markdown-файлах с обязательными полями, либо в отдельных служебных файлах рядом с ними.

## Что НЕ в скоупе

- Реализация repository/service слоя
- Генерация конспекта
- UI файлового навигатора и редактора
