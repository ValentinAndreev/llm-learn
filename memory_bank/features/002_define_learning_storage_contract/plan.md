# Контракт хранения учебных артефактов — план реализации

## Обзор подхода

Это docs-only фича внутри новой корневой директории `learning/`. Сначала создаём сами директории, затем записываем в `learning/process.md` контракт хранения и workflow, после чего отдельно проверяем, что документ покрывает все требования спеки.

## Шаги

### 1. Создать корневую структуру `learning/`
**Файлы:** `learning/process.md` (новый), `learning/prompts/.gitkeep` (новый), `learning/topics/.gitkeep` (новый)
**Что делаем:** Создать директорию `learning/` и сразу зафиксировать в git две рабочие поддиректории `learning/prompts/` и `learning/topics/` через `.gitkeep`, чтобы контракт ссылался на реально существующую структуру.
**Проверка:** В дереве проекта существуют `learning/`, `learning/prompts/` и `learning/topics/`.

### 2. Описать границу между БД и файловой структурой
**Файлы:** `learning/process.md` (изменить)
**Что делаем:** В начале `learning/process.md` описать назначение директории `learning/`, зафиксировать, что `Dialog`, `Message` и workflow-поля остаются в БД, а prompt templates и файловые артефакты темы живут под `learning/`.
**Проверка:** В `learning/process.md` есть отдельный раздел с явным разделением зон ответственности БД и файлового слоя.

### 3. Зафиксировать дерево каталогов и lifecycle темы
**Файлы:** `learning/process.md` (изменить)
**Что делаем:** Описать каталоги `learning/prompts/` и `learning/topics/`, момент создания `learning/topics/<topic_slug>/`, а также обязательные файлы и директории первой версии темы: `meta.yml`, `brief.md`, `conspect_prompt.md`, `conspects/current.md`, `conspects/versions/`, `reviews/`, `reviews/history/`.
**Проверка:** В `learning/process.md` перечислены все обязательные файлы и директории из спеки.

### 4. Зафиксировать формат `meta.yml` и правила `topic_slug`
**Файлы:** `learning/process.md` (изменить)
**Что делаем:** Описать обязательные ключи `meta.yml` (`topic_slug`, `source_dialog_id`, `created_at`, `updated_at`, `current_state`) и правила формирования `topic_slug`.
**Проверка:** В `learning/process.md` есть отдельный блок с форматом `meta.yml` и отдельный блок с правилами `topic_slug`.

### 5. Добавить примеры корректного и некорректного состояния
**Файлы:** `learning/process.md` (изменить)
**Что делаем:** Добавить пример корректной структуры conspect-темы и пример некорректного состояния, который нарушает контракт.
**Проверка:** В `learning/process.md` есть два явных примера: корректный и некорректный.

### 6. Финальная проверка спеки и текущего проекта
**Файлы:** `learning/process.md` (проверка), `memory_bank/features/002_define_learning_storage_contract/spec.md` (проверка)
**Что делаем:** Сверить `learning/process.md` с требованиями спеки, затем прогнать существующие тесты проекта, чтобы docs-only изменение не сопровождалось случайными правками кода.
**Проверка:** `bundle exec rspec` завершается без ошибок.

## Зависимости между шагами

- Шаг 2 требует шага 1, потому что `learning/process.md` должен лежать в уже созданной директории `learning/`
- Шаги 3–5 требуют шага 2, потому что сначала нужно зафиксировать верхнеуровневое назначение `learning/`, а затем наполнять его деталями
- Шаг 6 выполняется последним, потому что сверяет итоговый документ с полной спекой
